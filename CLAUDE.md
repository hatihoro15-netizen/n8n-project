# CLAUDE.md — 운영 지침 (4파일 체계: HANDOFF / process / PROGRESS)

> ⚠️ 이 문서는 프로젝트 운영의 "최상위 규칙"입니다. 임의로 자동 수정하지 말고, 변경이 필요하면 사용자에게 먼저 변경 이유/영향을 보고한 뒤 반영합니다.

---

## 0) 파일 역할(절대 혼동 금지)
- CLAUDE.md: **행동 지침 / 우선순위 / 보고 형식 / 업데이트 규칙** (변하지 않는 운영 규정)
- HANDOFF.md: **다음 세션 즉시 실행 스냅샷** (항상 Overwrite / 최신 정답)
- process.md: **정제된 지식베이스** (컨벤션, ADR(결정 이유), 프로젝트 고유 트러블슈팅만 Append)
- PROGRESS.md: **진행 일지(히스토리)** (날짜별 Append / 시도·결과·경과 기록)

### 언어 규칙
- **한국어로만 소통한다.** 코드/커맨드/파일명은 영문 유지, 설명/보고/질문은 모두 한국어.

### "정답(소스 오브 트루스)" 규칙
- **Next Actions / Last Run / Blockers의 최신 정답은 항상 HANDOFF.md**이다.
- PROGRESS.md의 Next는 "방향/메모" 수준까지만 허용한다. **실행 커맨드/즉시 지시는 HANDOFF에만** 둔다.

## 0-1) 각 파일 상단 경고 문구(필수로 삽입)

### HANDOFF.md 상단
> ✅ 이 파일은 "최신 스냅샷"이 목적이라 매번 전체 Overwrite가 정석입니다. (과거 기록은 PROGRESS.md에 남기고, 여기엔 최신 상태/Next/LastRun/Blockers만 유지)

### process.md 상단
> ❌ 이 파일은 프로젝트 지식베이스입니다. Overwrite/삭제 금지(append만). 정리(압축)가 필요해도 ADR/트러블슈팅 핵심은 남기고, 중복만 줄입니다.

### PROGRESS.md 상단
> ❌ 이 파일은 "진행 일지(히스토리)"입니다. 날짜별 기록은 append만(삭제/overwrite 금지). (선택) 상단 요약 섹션이 있다면 그 부분만 최신으로 갱신 가능.

## 4) 작업 방식(큰 프로젝트용 운영 규칙)

### 4-1) 최소 변경 원칙
- 필요 없는 리팩토링/추상화/새 의존성 추가 금지.
- 새 의존성/아키텍처 변경이 필요하면:
  - (1) 이유 1~2줄 → (2) 대안 1개 → (3) 영향/리스크 1개를 먼저 적고 진행.

### 4-2) 보안/비밀값 절대 규칙
- 토큰/키/계정/내부URL 등 비밀값은 절대 출력/커밋/로그에 남기지 않는다.
- 예시는 항상 더미값/환경변수로 대체한다.
- 단, CLAUDE.md의 "이 레포의 기본 정보"에 기재된 VPS/n8n 정보는 프로젝트 운영을 위한 고정 정보로 예외로 허용한다(그 외 내부 URL/키/토큰은 금지).

### 4-3) 비용/외부 API 규칙
- 유료 API 호출은 불필요한 반복 최소화(캐시/샘플/로컬 검증 우선).
- "한 번에 끝내기"보다 "작게 검증 → 확장" 순서로 진행.

### 4-4) 막히면 추측 금지(디버깅 프로토콜)
재현 단계 → 원인 후보 2~3개 → 최소 실험 1개 → 결론 순으로 진행한다.
- 해결되면 process.md에 "근본 원인/해결/예방"까지 정제해서 기록한다.

## 6) 파일 업데이트 규칙(핵심)

### 6-1) HANDOFF.md 업데이트(스냅샷 / Overwrite)
아래 상황이면 **스스로 HANDOFF 업데이트 초안**을 제안한다:
- 큰 기능 1개 완료
- 중요 이슈 해결/중단/보류
- 서버/배포/환경이 바뀜
- "다음 세션에서 바로 해야 할 일"이 바뀜

HANDOFF 규칙:
- HANDOFF.md는 **항상 전체 Overwrite**(로그처럼 누적 금지)
- HANDOFF.md에는 반드시 포함:
  - Current Status / Goal
  - Next Actions(우선순위 1~3, 가능하면 커맨드 포함)
  - Blockers / Known Issues
  - Last Run Command + Result + 실행 위치(Local/VPS)
  - 변경 파일(경로) 또는 워크플로우 ID
- 서버 실행/테스트가 한 번이라도 성공하면, 즉시 HANDOFF.md의 `Last Run Command`와 `Result`를 최신 값으로 채운다.

### 6-2) process.md 업데이트(정제된 지식만 / Append)
아래에 해당하면 process.md에 "항목 1개"로 추가(Append)한다:
- 환경/설정 변경(버전, ENV, 브랜치/배포 방식, 새 의존성)
- 프로젝트 고유 버그/재현 어려운 이슈의 **근본 원인 + 해결책 + 예방**
- 반복되는 작업 패턴/컨벤션 확정
- 의사결정(ADR): "왜 이 방식을 선택했는지"가 재발할 것 같은 경우

오염 방지:
- 단순 문법/오타/일회성 에러/누구나 아는 일반론은 기록 금지.

### 6-3) PROGRESS.md 업데이트(일지 / Append)
PROGRESS는 "히스토리"다. 날짜별로 Append만 한다.
- 기록 대상: Done / Tried / Result / Next(방향만) / Files·Links / Notes
- 기록 금지: 실행 커맨드의 '정답'(그건 HANDOFF로)
- 압축 규칙(선택): 너무 길어지면 "중복/잡담/반복"은 줄이고,
  '결정/전환점/핵심 버그 해결/중요 변경' 중심으로 유지한다.

권장 템플릿:
```
## YYYY-MM-DD
### ✅ Done
- [x] ...
### 🔁 Tried
- [ ] ...
### 📌 Result
- ...
### ➡️ Next (방향만 / 실행 커맨드는 HANDOFF)
- ...
### 📁 Files / Links
- ...
```

## 7) VPS / n8n 작업 시 추가 강제 규칙(해당 시)
- VPS/서버 작업 전 반드시 백업 여부 확인 후 진행한다.
- n8n 워크플로우 변경 전 반드시 기존 워크플로우 Export(JSON) 백업을 먼저 수행한다.
- **n8n 워크플로우 수정 시 top-level nodes와 activeVersion.nodes 양쪽 다 수정할 것.**
- **직렬 노드 삽입 시 이후 노드의 $json 참조가 깨지는지 반드시 확인할 것.**
- **Google Sheets update 노드는 alwaysOutputData: true 필수.**
- IP/호스트/경로 하드코딩 금지(환경변수 사용).
- 변경 후 가능하면 품질 체크/스모크 테스트를 실행하고 결과를 남긴다.

## 8) "큰 프로젝트" 운영 팁(선택이지만 강력 추천)
- 작업 단위를 "작게" 쪼개서 HANDOFF의 Next Actions(1~3)로만 운영한다.
- PROGRESS에는 "왜/무엇/결과"만 짧게 남기고, 규칙으로 굳어지면 process(ADR/트러블슈팅)로 승격한다.
