# 주요 작업 흐름

## 워크플로우 생성
1. 기획서 확인 (docs/n8n_기획서_최종.xlsx)
2. 기존 워크플로우 참조 (workflows/*.json)
3. 프롬프트 작성 (workflows/*_prompt.md)
4. 워크플로우 JSON 생성
5. quality-check.sh 실행
6. n8n API로 업로드
7. **에디터 F5 새로고침**
8. 테스트 실행 및 확인
9. PROGRESS.md 업데이트

## 워크플로우 수정
1. 수정 대상 워크플로우 JSON 읽기
2. 관련 프롬프트 파일 확인
3. **git commit (수정 전 상태 저장)**
4. 필요한 부분만 수정 (파일 전체 교체 금지)
5. quality-check.sh 실행
6. n8n API로 업로드
7. **에디터 F5 새로고침**
8. 동작 확인
9. PROGRESS.md 업데이트

## 프롬프트 수정
1. 해당 *_prompt.md 읽기
2. 캐릭터 설정/대사 규칙 확인
3. **git commit (수정 전 상태 저장)**
4. 프롬프트 텍스트 수정
5. 워크플로우 JSON의 해당 노드에 반영
6. 테스트 실행
7. PROGRESS.md 업데이트

## 워크플로우 삭제
1. **반드시 사용자 확인 받기**
2. git commit (삭제 전 상태 저장)
3. n8n에서 비활성화
4. 로컬 파일 삭제
5. PROGRESS.md 업데이트

---

## 의사결정 규칙

### 작업 중단 조건 (반드시 질문하고 확인 받기)
- 기존에 동작하던 기능을 수정해야 할 때
- 파일 삭제가 필요할 때
- 두 가지 방법 중 선택이 불확실할 때
- 요구사항이 모호할 때
- 예상치 못한 에러가 발생했을 때

### 판단 기준
- 확실하지 않으면 작업 중단하고 질문
- 임시방편 금지, 근본 원인 해결
- 단순한 방법이 있으면 복잡한 방법 선택 금지
- 기존 코드 스타일 유지

## 롤백 전략

### 작업 전 필수
- 반드시 git commit 후 작업 시작
- 커밋 없이 수정 시작 금지

### 문제 발생 시
1. 즉시 작업 중단
2. git diff로 변경사항 확인
3. git revert 또는 git checkout으로 롤백
4. 롤백 후 원인 파악
5. 원인 파악 완료 후에만 재시도
6. 같은 방법으로 재시도 금지 (다른 접근 방법 사용)

### 롤백 후 보고서 작성
- 무엇이 잘못됐는지
- 왜 잘못됐는지
- 다음에는 어떻게 할 것인지
- docs/06-error-patterns.md에 패턴 추가
